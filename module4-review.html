Claude Projects - Building Knowledge Bases

Quick Reference Guide

This guide summarizes the key concepts from Module 4: what Projects are, how they differ from Skills and Custom Instructions, building effective knowledge bases, collaborating through shared Projects, and maintaining them over time.

What Are Claude Projects?

The Core Concept

Projects are workspaces where Claude has persistent access to background knowledge, documents, and custom instructions. Unlike regular conversations that start fresh each time, Projects maintain context across all conversations within that workspace.

The Analogy: A Project is like having a dedicated workspace with a file cabinet. Every conversation you have in that Project has access to everything in the file cabinet.

The Core Problem Projects Solve

Without Projects: You re-upload documents and re-explain background in every new conversation.

With Projects: Claude already has your documents, previous conversations, and context automatically.

What Projects Contain

  • Uploaded documents (PDFs, Word, Excel, PowerPoint, text files, CSVs, images)
  • Custom instructions specific to this Project
  • Links and references
  • All conversation history within the Project

Important: Project knowledge is accessible to ALL conversations in that Project, but conversation history is specific to each chat thread.

When to Use Each

Custom Instructions: Communication style and preferences that apply everywhere

Skills: Repeatable task workflows that happen across different contexts

Projects: Ongoing work requiring accumulated context and collaboration

Building Effective Knowledge Bases

What Makes Good Project Knowledge

  • Focused and Relevant: Upload the 5-10 documents directly relevant to this initiative, not your entire company wiki
  • Well-Organized: Use clear, descriptive file names (not "notes.pdf" or "final-v3.docx")
  • Current and Maintained: Remove outdated documents when plans change
  • Actionable: Upload executive summaries or create summary documents, not 50-page reports
Rule of Thumb:

If you wouldn't expect a new team member to read it in their first week on the project, don't upload it to the Project.

Three Organization Strategies

Strategy 1: Create a Project README

Upload a document that maps what's in the Project and how everything fits together. Include: Project Overview, Key Documents (with descriptions), Key Dates, Decision Log.

Strategy 2: Use Clear File Names

Instead of "document (1).pdf" use "2025-01-15-stakeholder-meeting-notes.pdf"

Strategy 3: Regular Maintenance

Monthly check-in to remove outdated documents, upload new ones, update README, archive completed items.

Sharing and Collaboration

How Project Sharing Works

  • Project owner: Full access (edit, delete, add/remove members)
  • Project members: Can chat in Project, access all knowledge, see all conversations
  • Everyone else: Cannot see the Project exists

Important: All conversations in a shared Project are visible to all members.

Pattern 1: Cross-Functional

  • Product, Engineering, Marketing share one Project
  • Contains strategy, requirements, timelines
  • Everyone sees cross-functional decisions

Pattern 2: Team Knowledge Base

  • Entire team shares one Project
  • Process docs, templates, best practices
  • Living wiki Claude can reference
  • Useful for onboarding

Pattern 3: Initiative-Specific

  • Project for specific 3-6 month initiative
  • Share with everyone on that initiative
  • Archive when complete
  • Keeps focus tight
Maintaining Projects Over Time

When to Update Project Knowledge

  • Strategy changes
  • New documents are created
  • Previous decisions are reversed
  • Timelines shift
  • New stakeholders join
  • Requirements evolve

üö© Red Flags: Your Project Needs Maintenance

  • Claude references outdated information
  • You find yourself correcting Claude frequently
  • Project has 50+ documents and you're not sure what half are
  • File names are confusing or duplicated
  • Custom Instructions no longer match current goals
  • Team members are confused about what's current

Archive vs. Delete

Archive when: Project is complete but you might reference it later, it's a good template for future work, it contains historical decisions you want to preserve.

Delete when: Project contains outdated information with no historical value, it was experimental and didn't work out, it's been fully transitioned to another system.

How to archive: Create final summary, document key decisions, remove temporary files, update README with "Project completed [date]", change name to "[ARCHIVED] [Original Name]"

Projects + Skills + Artifacts = Power

Each Tool Has a Role

  • Projects: "What's the background and context?"
  • Skills: "How should I do this task?"
  • Custom Instructions: "How should I communicate?"
  • Artifacts: "How should I present the output?"

Together they create: Contextually aware, properly formatted, brand-compliant outputs that save massive time.

Example Workflow: Product Launch

Setup: Project with strategy docs + Brand Guidelines Skill + Your Custom Instructions

Ask: "Create a launch readiness presentation for executives using our brand guidelines"

What happens:

  • Claude loads Project knowledge (strategy, research, timeline)
  • Claude activates Brand Guidelines Skill (colors, logos)
  • Claude applies your Custom Instructions (your style)
  • Claude creates an Artifact (interactive presentation)

Result: Branded presentation with accurate project context

Best Practices for Shared Projects

‚úÖ Five Key Practices

  • Maintain Clear Structure: Use README so everyone knows what's in the Project
  • Document Decisions: Update README or pin decisions in conversations
  • Keep It Current: Assign someone to remove outdated docs, add new info, update README
  • Use Clear Chat Titles: "Customer Feedback Analysis - Jan 2025" not "Chat 1"
  • Set Expectations: Align on what Project is for, update frequency, who maintains it

üéØ Key Takeaway:

Projects maintain persistent context across conversations through uploaded documents and custom instructions. Different from Skills (task procedures) and Custom Instructions (global preferences). Build effective knowledge bases by staying focused (5-10 relevant docs), organizing clearly (README + descriptive names), and maintaining regularly (remove outdated, add new). Share Projects for collaboration but remove sensitive content first. Combine Projects + Skills + Artifacts for powerful, context-aware workflows.
